stat_item_count.cpp：
    玩家背包表 player_bag 数据量达到10亿行，你选择了 player_id % 128 做哈希分表。现在运营需要按“物品ID”统计全服持有量，如何在不影响在线服务的情况下高效完成？请给出方案及C++伪代
    核心背景
        分表规则：player_bag_0 ~ player_bag_127，共 128 张分表，路由规则 shard_id = player_id % 128
        约束：不影响在线服务（禁止锁表、禁止全表扫描主库、禁止高耗时查询）
        目标：统计全服所有物品 ID 的总持有量（sum (物品数量)，去重持有数可按需调整）
    方案设计思路
        隔离在线流量：从 ** 从库（只读库）** 读取数据，绝对不影响主库在线服务
        并行扫描：128 张分表独立并行查询，利用多核 / 多线程提升效率
        轻量级查询：每张表只查询 item_id, count 核心字段，无多余数据
        内存聚合：用哈希表在内存中累加所有物品的总数量，避免多次 IO
        可控压力：限制并发线程数（如 16~32 线程），防止从库带宽 / CPU 打满
        结果落地：聚合完成后写入结果表，供运营使用
    方案优势
        无锁、无主库操作，100% 不影响在线服务
        并行处理，128 张表可在分钟级完成统计（远快于单表全扫描）
        内存聚合效率极高，无中间表 IO 开销
        线程数可配置，灵活控制数据库压力
mysql_redis_Data Consistency:
    某游戏充值后需要更新玩家钻石数（MySQL + Redis缓存）。请设计一个保障最终一致性的更新流程，并指出在C++服务中如何避免“缓存与数据库不一致”的典型陷阱（如并发写、删除失败等）。给出带重试/补偿机制的代码。
    
    一致性原则
        1. 先更新数据库，再删除缓存（绝对不能先更新缓存！）
        2. 失败必须重试 / 补偿
        3. 并发写必须加锁
    标准安全流程（最终一致性）
        玩家维度加锁（防止并发写覆盖）
        查询当前钻石（加锁后查，保证最新）
        更新 MySQL 钻石（事务保证）
        删除 Redis 缓存（让下次读自动重建）
        删除失败 → 重试 + 异步补偿
        解锁  
    为什么是删除缓存不是更新缓存？
        因为并发场景下，更新缓存一定会导致脏数据。删除是最安全的策略。
    如何避免缓存与数据库不一致
        坑 1：并发写导致钻石覆盖（A、B 同时加钻石）
            解决方案：玩家维度加互斥锁同一玩家同时只能有一个写操作，从根源杜绝并发覆盖。
        坑 2：先更新缓存，再更新 DB（必不一致）
            正确：先更新 DB，再删缓存
        坑 3：Redis 删除失败
            解决方案：重试 3 次 + 异步补偿任务
            最终一定能删成功，保证最终一致。
        坑 4：查询时缓存旧值
            解决方案：缓存加过期时间（自动兜底）
            即使删失败，最多脏数据 1 天，自动恢复。
